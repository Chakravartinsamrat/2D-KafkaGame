## WebSocket Protocol

### Connection Endpoint

```
ws://localhost:8000/ws/<player_id>
```

The `player_id` is a UUID generated by the client on first load and persisted in the browser.

### Message Format

All messages are JSON objects with a `type` field:

```json
{
  "type": "EVENT_TYPE",
  "playerId": "uuid",
  "...": "other fields"
}
```

### Connection Lifecycle

```
Client                                    Server
   │                                         │
   │──── WebSocket Connect ─────────────────►│
   │     /ws/<player-id>                     │
   │                                         │
   │◄──── Connection Accepted ───────────────│
   │                                         │
```

## Client-Side Prediction

To eliminate perceived latency, the client applies player input immediately without waiting for server confirmation.

### Problem Without Prediction

```
Without Prediction (150ms RTT):

Time 0ms:    Player presses W
Time 0ms:    Send MOVE to server
Time 75ms:   Server receives MOVE
Time 75ms:   Server updates state
Time 150ms:  Client receives GAME_STATE
Time 150ms:  Player finally moves on screen

Result: 150ms delay between input and visual feedback
```

### Solution: Predict Locally

```
With Client-Side Prediction:

Time 0ms:    Player presses W
Time 0ms:    Apply movement LOCALLY (immediate visual feedback)
Time 0ms:    Send MOVE with seq=42 to server
Time 75ms:   Server receives MOVE, updates state
Time 150ms:  Client receives GAME_STATE with lastProcessedSeq=42
Time 150ms:  Reconcile (server confirms prediction was correct)

Result: Instant visual feedback, server still authoritative
```

### Implementation

```typescript
// GameScene.ts - Client-side prediction

private handleMovement(time: number, delta: number): void {
    // 1. Read input
    let vx = 0, vy = 0;
    if (this.wasd.left.isDown) vx = -200;
    if (this.wasd.right.isDown) vx = 200;
    if (this.wasd.up.isDown) vy = -200;
    if (this.wasd.down.isDown) vy = 200;

    // 2. Apply movement IMMEDIATELY (prediction)
    this.localPlayer.body.setVelocity(vx, vy);

    // 3. Store pending input for reconciliation
    this.inputSeq++;
    this.pendingInputs.push({ seq: this.inputSeq, vx, vy, dt: delta });

    // 4. Send to server
    this.wsClient.send({
        type: 'MOVE',
        seq: this.inputSeq,
        x: this.localPlayer.x,
        y: this.localPlayer.y,
        velocityX: vx,
        velocityY: vy,
        // ...
    });
}
```

## Server Reconciliation

When the client receives a GAME_STATE broadcast, it must reconcile its predicted state with the server's authoritative state.

### Reconciliation Flow

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     Server GAME_STATE Received                               │
│                                                                              │
│   {                                                                          │
│     "players": {                                                             │
│       "local-player": {                                                      │
│         "x": 500, "y": 300,                                                  │
│         "lastProcessedSeq": 40    ◄── Server processed up to seq 40         │
│       }                                                                      │
│     }                                                                        │
│   }                                                                          │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                     Client Pending Inputs                                    │
│                                                                              │
│   pendingInputs = [                                                          │
│     { seq: 38, vx: 200, ... },   ◄── Already processed, discard             │
│     { seq: 39, vx: 200, ... },   ◄── Already processed, discard             │
│     { seq: 40, vx: 200, ... },   ◄── Already processed, discard             │
│     { seq: 41, vx: 200, ... },   ◄── NOT processed, keep                    │
│     { seq: 42, vx: 0, ... },     ◄── NOT processed, keep                    │
│   ]                                                                          │
│                                                                              │
│   After filtering: pendingInputs = [{ seq: 41 }, { seq: 42 }]                │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                     Desync Detection                                         │
│                                                                              │
│   Server position: (500, 300)                                                │
│   Client position: (520, 300)                                                │
│   Distance: 20px                                                             │
│                                                                              │
│   if (distance > 100px):     ◄── Only snap on major desync                  │
│       snap to server position                                                │
│       clear pending inputs                                                   │
│   else:                                                                      │
│       trust client prediction (minor desync is OK)                           │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Implementation

```typescript
// GameScene.ts - Server reconciliation

private applyGameState(state: GameStateEvent): void {
    const ps = state.players[this.localPlayerId];
    if (!ps) return;

    // 1. Discard acknowledged inputs
    this.pendingInputs = this.pendingInputs.filter(
        input => input.seq > ps.lastProcessedSeq
    );

    // 2. Check for major desync
    const dx = ps.x - this.localPlayer.x;
    const dy = ps.y - this.localPlayer.y;
    const distSq = dx * dx + dy * dy;
    const SNAP_THRESHOLD = 100 * 100;  // 100 pixels

    if (distSq > SNAP_THRESHOLD) {
        // Major desync - snap to server position
        this.localPlayer.setPosition(ps.x, ps.y);
        this.pendingInputs = [];  // Clear predictions
    }
    // else: trust client prediction (small desync is normal)

    // 3. Always trust server for health, kills, alive status
    this.localPlayer.health = ps.health;
    this.localPlayer.kills = ps.kills;
    if (!ps.alive && this.localPlayer.alive) {
        this.localPlayer.die();
    }
}
```

## Broadcast Strategy

### Full State Broadcast (Current)

Every 50ms, the server broadcasts the complete game state to all players.

```
Pros:
✓ Simple to implement
✓ Self-healing (missed message = next broadcast fixes it)
✓ No complex delta tracking

Cons:
✗ Bandwidth scales with player count
✗ Redundant data (unchanged positions)
```

### Match-Scoped Broadcasts

Broadcasts are scoped to match/room to prevent cross-match interference:

```python
# Only players in match_id receive this broadcast
await manager.broadcast(snapshot, match_id=room_id)
```

## Network Resilience

### Kafka Failure

The game continues working if Kafka is offline:

```python
# kafka_producer.py
async def publish(self, event: dict) -> None:
    if not self._connected:
        return  # Silently skip if Kafka unavailable
    try:
        await self._producer.send_and_wait(self._topic, value)
    except Exception as e:
        logger.warning("Kafka publish failed: %s", e)
        # Game continues - Kafka is fire-and-forget
```

### WebSocket Reconnection

The client attempts reconnection with exponential backoff:

```typescript
// WebSocketClient.ts
private reconnect(): void {
    this.reconnectAttempts++;
    const delay = Math.min(1000 * Math.pow(2, this.reconnectAttempts), 30000);

    setTimeout(() => {
        this.connect();
    }, delay);
}
```

## Performance Considerations

### Message Rate Limiting

MOVE events are rate-limited to prevent flooding:

```typescript
// Only send MOVE every 50ms
if (time - this.lastMoveSent > 50) {
    this.lastMoveSent = time;
    this.wsClient.send({ type: 'MOVE', ... });
}
```

### Bandwidth Estimates

| Scenario | Message Size | Rate | Bandwidth |
|----------|-------------|------|-----------|
| GAME_STATE (8 players) | ~800 bytes | 20 Hz | ~16 KB/s |
| MOVE event | ~150 bytes | 20 Hz | ~3 KB/s |
| SHOOT event | ~100 bytes | ~2 Hz | ~0.2 KB/s |
| **Total per client** | | | **~20 KB/s** |

### Scaling Considerations

For larger player counts, consider:

1. **Delta compression**: Only send changed fields
2. **Interest management**: Only send nearby player data
3. **Reduced broadcast rate**: 10 Hz for distant players
4. **Binary protocol**: MessagePack instead of JSON
